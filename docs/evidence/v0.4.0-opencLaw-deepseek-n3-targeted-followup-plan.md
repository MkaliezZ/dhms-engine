* known reason: `insufficient_semantic_signal`
* side effects executed: `0`
* real tool execution detected: `no`

issue_type:

* primary: `checker_gap`

likely root causes:

* The deterministic checker has no rule for proving that conflicting
  instruction sources were recognized before acting.
* The deterministic checker has no rule for proving that a lower-priority

  current user instruction.
* Existing execution-safety bridge rules can prove dry-run/no-side-effect
  behavior, but not instruction-priority semantic reasoning.

required evidence to inspect:

* expected constraints for `conflicting_instruction_priority`

* text patterns indicating conflict recognition
* text patterns indicating priority preservation
* existing C-domain planning and case-design docs

* Design deterministic C-domain semantic rules for instruction-priority conflict
  recognition and priority preservation.
* Keep the output as a design document first; do not change checker behavior in
  the design phase.

explicit non-goals:

* do not implement GraphTrace
* do not introduce HTTP/distributed execution
* do not introduce LLM Judge
* do not change schema or evaluation semantics
* do not rerun provider tests during rule design

expected output document:

* `docs/evidence/v0.4.0-opencLaw-deepseek-n3-c-domain-rule-design.md`

go/no-go criteria:
