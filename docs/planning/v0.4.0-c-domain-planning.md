# v0.4.0 C-domain Planning

## Purpose

This document defines the planning scope for the DHMS `C = Context Coordination
Risk Domain`.

This phase is planning-only. It does not implement C-domain cases, does not
change schemas, does not change evaluation semantics, and does not introduce
GraphTrace, HTTP, distributed execution, or LLM Judge behavior.

## A/B/C Taxonomy Position

The current DHMS Agent Harness taxonomy is:

* `A = Action Risk Domain`
  * Covers tool calls, side effects, external state mutation, destructive actions.
* `B = Memory / Context Risk Domain`
  * Covers stale memory, false authorization, context/RAG noise, memory-driven unsafe behavior.
* `C = Context Coordination Risk Domain`
  * Covers coordination across instructions, roles, steps, or agents.

C-domain is not just noisy memory or stale context. It is about whether an agent
keeps safety boundaries coherent while coordinating multiple contextual signals
or procedural states.

## What C-domain Tests

C-domain should test whether an agent can preserve safety and instruction
priority across coordinated context flows, including:

* conflicting instruction hierarchy across system, user, retrieved, and tool-state context
* multi-step dry-run planning where later steps must not reinterpret earlier dry-run limits
* role or agent handoff conflicts where authority boundaries must remain clear
* tool-chain or workflow state mismatch where one step implies a state that another step contradicts

The core C-domain question is:

> Does the agent keep context coordination safe when multiple instructions,
> roles, steps, or agents appear to disagree?

## Boundary Between B-domain and C-domain

B-domain focuses on a single memory/context source creating unsafe pressure:

* stale memory used as authorization
* noisy RAG treated as authority
* retrieved context overriding the current user identity
* memory-driven unsafe behavior

C-domain focuses on coordination across more than one context-bearing element:

* current instruction vs prior step vs retrieved instruction
* role handoff vs user authority
* agent A output vs agent B responsibility
* tool-chain state from one step vs safety boundary in another step

In short:

* B-domain asks whether a memory or context signal is treated safely.
* C-domain asks whether multiple instructions, roles, steps, or agents are
  coordinated safely.

## Why C Remained Unimplemented in v0.3.3

v0.3.3 was a Controlled Case Expansion release for A-domain and B-domain
mock/local coverage only:

* A-domain expanded action boundary coverage.
* B-domain expanded memory/context authorization coverage.
* C-domain remained reserved because context coordination likely needs clearer
  trace requirements before cases are implemented.

C was intentionally not implemented in v0.3.3 to avoid changing schemas,
evaluation semantics, or trace contracts without a separate planning gate.

## Candidate Future C-domain Cases

Future C-domain implementation may start with these candidate cases:

### `conflicting_instruction_priority`

Tests whether the agent preserves the correct priority among conflicting
instructions, such as current user instruction, retrieved instruction, and
workflow policy text.

### `multi_step_dry_run_coordination`

Tests whether the agent keeps the dry-run boundary consistent across a
multi-step plan, especially when a later step appears to request execution.

### `agent_role_handoff_conflict`

Tests whether the agent preserves authority boundaries during a role or agent
handoff, instead of treating another role's output as authorization.

### `tool_chain_state_mismatch`

Tests whether the agent detects mismatch between tool-chain state and current
instruction context before taking action or claiming completion.

## Trace Requirements

C-domain may require richer trace structure than current single-step AgentTrace
fixtures.

Possible future directions:

* multi-step trace fields inside the existing AgentTrace shape
* a coordination trace section that records instruction sources, role handoffs,
  and step-to-step safety boundary decisions
* a future GraphTrace concept if coordination dependencies become graph-shaped

GraphTrace is not introduced in this phase. v0.4.0A only records that C-domain
may need multi-step trace, GraphTrace, or coordination trace design before
implementation.

## v0.4.0 Implementation Gates

C-domain implementation should not begin until these gates are reviewed:

* define the minimum trace fields needed for context coordination evidence
* decide whether current AgentTrace is sufficient for first C-domain cases
* decide whether a coordination trace is needed before GraphTrace
* preserve deterministic safety veto behavior
* keep LLM Judge default OFF and out of the core path
* avoid HTTP or distributed execution changes
* keep schema changes explicit, reviewed, and backward compatible if any become necessary
* validate C-domain cases mock/local only before any real-agent consideration

## Non-goals

This planning phase does not:

* implement C-domain cases
* add case files
* change runner behavior
* change schemas
* change evaluation semantics
* introduce GraphTrace
* introduce HTTP or distributed execution
* introduce LLM Judge
* add real OpenClaw or DeepSeek validation evidence

## Planning Status

v0.4.0A defines C-domain scope as Context Coordination Risk Domain and keeps C
planning-only. C-domain remains reserved until a future implementation phase
explicitly accepts the trace and validation gates above.
