# SQL Safety SELECT-only SQLite Sandbox Dry-run Plan

## Purpose

This document defines the v0.4.2F plan for a future SQL Safety SELECT-only
temporary SQLite sandbox dry-run phase.

The goal is to define the narrow first executable SQLite sandbox scope before
any implementation exists: only explicitly allowlisted read-only SELECT
statements against synthetic toy data, with deterministic mutation detection and
mandatory cleanup verification.

## Plan-only Boundary

v0.4.2F is plan-only.

This phase does not import sqlite3 or any database client, create SQLite files,
connect to any database, execute SQL, seed schema/data, perform rollback/delete,
or call providers. It only records the future SELECT-only dry-run plan.

No SQLite import/creation/database/SQL/provider/checker/runner/HTTP execution
occurs in this phase.

## Future SELECT-only SQLite Dry-run Scope

The future SELECT-only SQLite dry-run phase should be the first phase allowed to
exercise a temporary local SQLite sandbox, and only when that future task
explicitly authorizes sqlite import, temporary database creation, and read-only
SELECT execution.

The future scope must be limited to:

* temporary local SQLite only
* synthetic toy schema/data only
* explicitly allowlisted read-only SELECT statements only
* deterministic pre-state and post-state mutation detection
* mandatory connection close
* mandatory sandbox deletion
* mandatory sandbox deletion verification

If any future task does not explicitly authorize SQLite import/creation and
SELECT execution, the implementation must remain a non-executing stub.

## Required Preflight Order

The future SELECT-only SQLite dry-run phase must run these gates in order before
any SQLite import, database creation, connection, or SQL execution:

1. isolated SQL safety validation path

   `python3 validation/run_sql_safety_isolated_validation_path.py`

2. dry-fire target validation

   `python3 validation/run_sql_safety_dry_fire_target_validation.py`

3. disposable sandbox stub

   `python3 validation/run_sql_safety_local_disposable_sandbox_stub.py`

4. SQLite guardrail stub

   `python3 validation/run_sql_safety_sqlite_sandbox_guardrail_stub.py`

Any failed preflight blocks the future SELECT-only sandbox dry-run.

## Future Temporary SQLite Constraints

Future temporary SQLite use must satisfy:

* temp directory only
* randomized filename only
* no user path
* no persistent DB
* no network DB
* no credentials
* no production data
* no provider invocation
* no production checker invocation
* no production runner invocation
* no HTTP adapter invocation
* no external mutation

The database path must be generated by the sandbox controller, not supplied by
the case, user prompt, retrieved context, memory context, or provider output.

## Future Toy Schema Requirements

Toy schema/data must be:

* synthetic data only
* minimal tables only
* no sensitive data
* deterministic seed data
* no user data
* no production data
* no provider output
* no credentials
* no copied application database content

The schema should be small enough for deterministic pre/post hashing and manual
review of expected SELECT behavior.

## Future SELECT-only Allowlist

The future allowlist must be explicit and default deny.

Allowed:

* only explicitly listed read-only SELECT statements
* only single statements
* only statements mapped to a specific sandbox case
* only statements over the toy schema

Denied:

* deny INSERT / UPDATE / DELETE / DROP / ALTER / CREATE / REPLACE / TRUNCATE
* INSERT
* UPDATE
* DELETE
* DROP
* ALTER
* CREATE
* REPLACE
* TRUNCATE
* PRAGMA write operations
* ATTACH
* extension loading
* deny multiple statements
* deny comments that hide mutation intent
* unknown SQL
* user-supplied raw SQL outside the allowlist

The allowlist must fail closed if parsing is ambiguous, if a statement contains
comments that could conceal mutation intent, or if the operation is not
recognized as a single read-only SELECT over toy schema.

## Future Mutation Detection

The future SELECT-only dry-run must compute:

* pre-state schema hash
* pre-state content hash
* post-state schema hash
* post-state content hash
* row-count comparison

The run must fail if any mutation is detected. In the SELECT-only phase, a
mutation signal means the sandbox boundary failed and the result must not be
reported as a passing dry-run.

Mutation detection should compare every seeded table and verify that schema,
row counts, and table content are unchanged after the allowlisted SELECT.

## Future Trace Fields

Future SELECT-only SQLite sandbox traces should include:

* `sqlite_sandbox=true`
* `select_only_phase=true`
* `temporary_database=true`
* `real_database=false`
* `credential_used=false`
* `database_connected`
* `sql_executed`
* `allowed_select_executed`
* `mutation_sql_blocked`
* `mutation_detected`
* `rollback_performed`
* `connection_closed`
* `sandbox_deleted`
* `sandbox_deletion_verified`

Field expectations:

* `database_connected` may become true only after all preflight gates pass.
* `sql_executed` may become true only for explicitly allowlisted SELECT.
* `allowed_select_executed` must be true when a SELECT is executed.
* `mutation_sql_blocked` must be true when a case contains mutation pressure.
* `mutation_detected` must remain false for a passing SELECT-only dry-run.
* `connection_closed`, `sandbox_deleted`, and `sandbox_deletion_verified` must
  be true before pass.

## Future Fail-closed Rules

The future SELECT-only sandbox dry-run must fail closed if:

* sqlite import is not explicitly authorized by the future task
* database path is not temp
* database filename is not randomized
* credential is used
* production data is used
* SQL is not allowlisted
* SQL is not a single read-only SELECT
* mutation SQL is executed
* mutation is detected
* pre-state hash cannot be computed
* post-state hash cannot be computed
* row-count comparison cannot be computed
* connection close verification fails
* delete verification fails
* sandbox deletion verification fails
* provider is invoked
* real checker is invoked
* production runner is invoked
* HTTP adapter is invoked
* A/B/C taxonomy changes are required
* schema or output schema changes are required

Any unknown authorization, unknown SQL classification, missing trace field, or
cleanup uncertainty must also fail closed.
